# Lab Result Entry (Structured Templates) — 2027 Modernization Plan

**Document type**: Implementation plan, following the same conventions as `reports/order-creation-v2-modernization-plan.md` (extract-don't-rewrite, phased delivery, explicit scope table, status column). Built on findings in [lab-result-templates-current-state-audit.md](lab-result-templates-current-state-audit.md) (what exists) and [lab-result-templates-problems-audit.md](lab-result-templates-problems-audit.md) (what's wrong).

**Why this exists now**: a Tanzania-standard structured Stool Analysis result form was requested to replace free-text entry. The audit found the structured system (`ResultTemplate` / `StructuredLabResultForm`) already exists and Stool Analysis is already fully modeled — what's needed is not a rebuild, it's fixing a live bug, closing one real field gap (Remarks/Impression), and hardening the data layer so this pattern scales cleanly to the rest of the Tanzanian standard test panel list. This plan is scoped as fixes and hardening on top of working code, not a rewrite.

## 1. Design decision: keep the existing `ResultTemplate` architecture

The `sections: [{ label, fields: [{code, label, type, options?}] }]` model, driven entirely from catalog `metadata.resultTemplate` JSON and rendered by one generic `StructuredLabResultForm.vue`, is the right shape for a modern LIS: adding a new test type's structured form is a data change (new catalog JSON), not a new Vue component. **Do not build a Stool-specific component** — that would regress the one part of this system that's already correctly generalized, and would need to be repeated for every future test type (Urinalysis, Malaria, HIV, Blood Group, Sputum, Widal already share this engine).

## 3. Non-goals

- Rebuilding `StructuredLabResultForm.vue` or the `ResultTemplate` type system from scratch — it's architecturally sound; the problems are data-duplication and one missing field type, not the design.
- A Stool-Analysis-specific Vue component — would break the one-engine-many-templates pattern already established for 7 test types.
- PDF/print-report generation, analytics/decision-support on structured results, or any consumer of `resultParameters` beyond the detail sheet — out of scope until the data itself is trustworthy (Phases 1-5).
- Touching the encounter-workflow result-entry path if one exists separately — this plan covers `laboratory-orders/IndexV2.vue`'s `LaboratoryStatusUpdateDialog.vue` only; confirm before Phase 1 whether encounter-context result entry shares this same dialog (it appears to, per the current-state audit, since there's only one `LaboratoryStatusUpdateDialog.vue` in the codebase) or has a separate surface that would also need the Phase 1/2 fixes.
